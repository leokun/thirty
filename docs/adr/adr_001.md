## ADR-001 : PostgreSQL vs MongoDB

**Date** : 2026-04-02
**Statut** : Accepté

### Contexte

Thirty a besoin d'une base de données pour stocker les utilisateurs, les repas loggés, la base d'aliments, et les scores calculés. Le projet précédent (iziprod) utilise MongoDB/Mongoose. La question se pose : rester sur du familier ou choisir l'outil adapté au domaine ?

### Options évaluées

**Option A — MongoDB**
- (+) Expérience existante forte (iziprod)
- (+) Flexibilité du schéma pour itérer vite sur la structure des aliments
- (+) Écosystème Mongoose mature avec NestJS
- (-) Les requêtes analytiques sont pénibles (agrégats semaine glissante, scores composites, COUNT DISTINCT sur 7 jours)
- (-) Les données sont fortement relationnelles : un FoodLog référence un Food, appartient à un MealEntry, qui appartient à un User
- (-) Pas de transactions ACID natives fiables pour les calculs de score multi-collections
- (-) Les jointures ($lookup) sont lentes et peu lisibles vs un SQL JOIN

**Option B — PostgreSQL**
- (+) Modèle relationnel naturel pour le domaine (User → MealEntry → FoodLog → Food)
- (+) Requêtes analytiques natives et performantes (window functions, GROUP BY sur date range, COUNT DISTINCT)
- (+) Transactions ACID pour la cohérence des scores
- (+) Excellent support JSON/JSONB si besoin de flexibilité ponctuelle (ex: profil microbiote d'un aliment)
- (+) Écosystème ORM TypeScript mature (Prisma, Drizzle, TypeORM)
- (-) Moins d'expérience directe (mais le SQL est connu)
- (-) Migrations plus rigides que le schéma-less de Mongo

### Décision

**PostgreSQL.**

Le domaine de Thirty est fondamentalement relationnel. La feature centrale — calculer un score de diversité sur une semaine glissante — est une requête analytique typiquement SQL :

```sql
SELECT COUNT(DISTINCT f.id)
FROM food_log fl
JOIN food f ON fl.food_id = f.id
WHERE f.is_plant = true
  AND fl.date >= CURRENT_DATE - INTERVAL '6 days'
  AND fl.user_id = :userId
```

En MongoDB, cette même requête nécessite un pipeline d'agrégation avec $lookup, $unwind, $match, $group — plus verbeux, plus lent, moins lisible.

De plus, le scoring quotidien implique des agrégats sur plusieurs axes simultanément, avec des pondérations — c'est le terrain de jeu naturel de SQL.

### Conséquences

- L'ORM choisi devra supporter PostgreSQL (Prisma et Drizzle le font nativement)
- Les profils microbiote des aliments pourront utiliser des colonnes JSONB pour garder de la flexibilité sur la structure interne
- La migration depuis un éventuel prototype MongoDB serait coûteuse → on part sur Postgres dès le jour 1
- Besoin de gérer les migrations de schéma proprement (outil intégré à l'ORM)
