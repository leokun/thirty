## ADR-008 : @thirty/core — clean architecture DDD par domaine

**Date** : 2026-04-02
**Statut** : Accepté
> Partiellement supercédé par [ADR-009](./adr_009.md) : types métier dans shared, adapters Prisma dans core, package @thirty/db centralisé.

### Contexte

Le package `@thirty/scoring` contenait ~50 lignes de fonctions pures de calcul (pondération, diversité, modificateurs de préparation). Il n'existait aucune couche d'orchestration pour assembler ces fonctions en use cases concrets : scorer un journal alimentaire, tracker la diversité hebdomadaire, générer des suggestions.

Sans cette couche, la logique d'orchestration aurait été dupliquée entre `apps/api` et `apps/web`.

Par ailleurs, les interfaces métier (`MicrobiomeProfile`, `PreparationModifier`, `DailyScoreBreakdown`) vivaient dans `@thirty/shared` alors qu'elles sont des concepts du domaine, pas des DTOs techniques.

### Options évaluées

**A) Garder `@thirty/scoring` séparé + ajouter `@thirty/core` plat**
- Pour : séparation des responsabilités, scoring reste indépendant
- Contre : scoring trop petit pour justifier un package (50 lignes), indirection artificielle (core wrappe scoring à 100%), graphe de dépendances plus complexe (shared → scoring → core → apps)

**B) Tout dans `@thirty/core` structuré en clean architecture DDD**
- Pour : un seul package métier, 4 domaines clairs, graphe simplifié (shared → core → apps), interfaces métier au bon endroit
- Contre : package plus gros (mais reste pur TS sans framework)

### Décision

**Option B** : un package `@thirty/core` structuré en clean architecture DDD avec 4 domaines.

```
packages/core/src/
├── domains/
│   ├── scoring/        # Fonctions pures de calcul (ex-@thirty/scoring)
│   ├── journal/        # Use case : scorer repas/journée
│   ├── diversity/      # Use case : diversité hebdomadaire, fenêtre glissante
│   └── suggestion/     # Use case : suggestions contextuelles
└── infrastructure/
    └── repositories/   # Repos in-memory pour les tests unitaires
```

Chaque domaine contient :
- `entities/` : objets d'identité du domaine
- `value-objects/` : objets immuables sans identité
- `repositories/` : interfaces pures (ports)
- `use-cases/` : orchestration métier
- `services/` : logique de domaine réutilisable

`@thirty/shared` ne garde que les enums (`FoodCategory`, `PreparationMethod`, etc.) et les constantes (`SCORING_WEIGHTS`, `WEEKLY_PLANT_GOAL`, `ROLLING_WINDOW_DAYS`).

### Conséquences

- `@thirty/scoring` est supprimé, son contenu migré dans `core/domains/scoring/`
- Les interfaces métier migrent de `shared` vers les value-objects des domaines concernés dans core
- Les tests unitaires des use cases utilisent les repos in-memory de `core/infrastructure/`
- Tests colocalisés (`*.test.ts` à côté du fichier testé), scope unitaire strict
- Les implémentations Prisma/NestJS des ports restent dans `apps/api/`
- Graphe de dépendances simplifié : `shared → core → apps`
