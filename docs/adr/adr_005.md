## ADR-005 : Mode de préparation comme multiplicateur de score

**Date** : 2026-04-02
**Statut** : Accepté

### Contexte

Un même aliment a un impact différent sur le microbiote selon son mode de préparation. Un chou cru, un chou cuit vapeur, et une choucroute (chou lactofermenté) ne nourrissent pas le microbiote de la même manière. Comment modéliser ça dans le scoring ?

### Options évaluées

**Option A — Aliments distincts par préparation**
- Créer des entrées séparées : "Chou cru", "Chou cuit", "Choucroute"
- (+) Simple conceptuellement — chaque entrée a son propre score fixe
- (-) Explosion combinatoire : 300 aliments × 10 modes = 3 000 entrées potentielles
- (-) Redondance massive dans la base
- (-) L'utilisateur doit chercher la bonne variante — friction de saisie
- (-) Maintenance cauchemardesque : changer le score de base du chou nécessite de modifier N entrées

**Option B — Aliment unique + multiplicateurs par préparation**
- Un aliment a un score de base, et chaque mode de préparation applique des facteurs multiplicateurs
- (+) Un seul "Chou" dans la base, avec des modificateurs pour chaque préparation
- (+) Le score final = score_base × facteur_préparation + bonus_microbiote
- (+) Maintenable : changer le score de base se répercute automatiquement
- (+) UX fluide : l'utilisateur cherche "chou", puis sélectionne le mode de préparation
- (+) Extensible : ajouter un nouveau mode de préparation = ajouter une ligne de facteurs
- (-) Les facteurs sont des approximations (mais c'est le cas dans toutes les approches)
- (-) Certains aliments n'existent que dans un mode (on ne mange pas de yaourt "frit")

**Option C — Hybride : multiplicateurs par défaut + overrides**
- Même principe que l'option B, mais avec la possibilité d'un override complet pour les cas où le multiplicateur ne suffit pas
- Ex : la choucroute n'est pas juste "chou × facteur_lactofermenté" — elle a aussi des souches bactériennes spécifiques (Leuconostoc, Lactobacillus) qui méritent un profil dédié
- (+) Meilleur des deux mondes : simplicité par défaut, précision quand nécessaire
- (-) Légèrement plus complexe à implémenter

### Décision

**Option C — Multiplicateurs par défaut avec overrides possibles.**

Le système de multiplicateurs couvre 90% des cas. Pour les 10% restants (aliments dont la version fermentée est fondamentalement différente du produit de base), on permet un override complet du profil.

Concrètement, dans le modèle :

```typescript
type PreparationModifier = {
  method: PreparationMethod;
  // Facteurs multiplicateurs (appliqués au score de base)
  fiberFactor: number;      // 0.0 - 2.0
  prebioticFactor: number;
  polyphenolFactor: number;
  probioticsFactor: number;
  microbiomeBonus: number;  // -2 to +5

  // Override optionnel (remplace le score de base au lieu de le multiplier)
  override?: Partial<MicrobiomeProfile>;
};
```

Exemple d'override : la choucroute utilise le profil de base du chou pour les fibres (× facteur), mais override le `probiotics_score` à 4/5 et ajoute un tag "Leuconostoc mesenteroides" — car ce n'est pas un simple "chou × quelque chose", c'est un aliment transformé avec un profil microbien propre.

### Implementation details

La formule simplifiee `score_base x facteur + bonus` est une bonne approximation, mais le code reel dans `apply-preparation-modifier.service.ts` presente quelques nuances :

1. **Fibres** : le score de fibres est la somme des fibres solubles et insolubles avant application du facteur : `fiber = (solubleFiberScore + insolubleFiberScore) x fiberFactor`
2. **omega3 et mucosalSupport** : ces deux axes ne sont pas affectes par le mode de preparation. Ils passent sans facteur multiplicateur (la preparation ne change pas la teneur en omega-3 ni le soutien muqueux).
3. **Bonus microbiome** : le `microbiomeBonus` n'est pas directement ajoute au score final lors du calcul par aliment. Il est stocke separement dans `ComputedFoodScore.bonus` et agrege plus tard via `aggregatePreparationScore()`.

### Conséquences

- La table `preparation_modifiers` a une colonne nullable `override_profile` (JSONB)
- Le moteur de scoring applique d'abord les facteurs, puis écrase avec l'override s'il existe
- La curation des aliments fermentés nécessite un travail spécifique pour identifier quels aliments méritent un override
- L'UX de saisie filtre les modes de préparation pertinents par aliment (pas de "yaourt germé")
- Les modes de préparation disponibles par aliment sont définis dans le profil de l'aliment (`available_preparations[]`)
