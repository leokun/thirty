## ADR-009 : Réorganisation des couches : types dans shared, adapters dans core, package db centralisé

**Date** : 2026-04-02
**Statut** : Accepté
> Supercède partiellement [ADR-008](./adr_008.md).

### Contexte

L'ADR-008 a posé les bases d'une clean architecture DDD dans `@thirty/core` avec 4 domaines, des ports purs, et des interfaces métier (`MicrobiomeProfile`, `PreparationModifier`, `DailyScoreBreakdown`) hébergées dans les value-objects de core.

Avec l'arrivée de la migration F1, trois limitations apparaissent :

1. **Types métier inaccessibles au frontend** : `apps/web` a besoin des types comme `MicrobiomeProfile` ou `DailyScoreBreakdown` pour typer ses composants et ses appels API. Importer `@thirty/core` depuis le frontend tire Prisma et toute l'infrastructure DB dans le bundle - ce n'est pas acceptable pour une PWA.

2. **Adapters Prisma isolés de leurs ports** : les implémentations Prisma des repositories vivent dans `apps/api/`, loin des interfaces qu'elles implémentent dans `core/domains/*/repositories/`. Si on remplace NestJS par un autre framework, il faut déplacer tous les adapters. Co-localiser ports et adapters dans core rend le domaine autonome.

3. **Prisma éparpillé** : le schema vit dans `apps/api/prisma/`, le seed dans `packages/food-db/`, le client est généré dans `apps/api/src/generated/prisma`. Trois emplacements pour un seul concern. Un package `@thirty/db` centralisé simplifie les migrations, le seeding et la génération du client.

### Options évaluées

**A) Statu quo (ADR-008 tel quel)**
- Pour : pas de migration, pas de risque de régression
- Contre : le frontend ne peut pas consommer les types métier sans tirer core+Prisma, les adapters restent découplés de leurs ports, Prisma reste fragmenté

**B) Exporter les types depuis core via un sous-path `@thirty/core/types`**
- Pour : pas besoin de déplacer les types, tree-shaking théorique
- Contre : en pratique les bundlers résolvent le package entier, Prisma se retrouve dans le graphe de dépendances du frontend, ne résout pas les problèmes d'adapters et de fragmentation Prisma

**C) Triple réorganisation : types dans shared, adapters dans core, package @thirty/db centralisé**
- Pour : le frontend importe `@thirty/shared` (leaf package, zéro dépendance lourde), les domaines de core sont autonomes (ports + adapters côte à côte), un seul endroit pour tout ce qui est Prisma
- Contre : migration non triviale (déplacement de fichiers, mise à jour des imports), ADR-008 partiellement inversé

### Décision

**Option C** : triple réorganisation.

#### 1. Types/interfaces métier migrent vers `@thirty/shared`

Les types suivants quittent les value-objects de `@thirty/core` pour `@thirty/shared` :
- `MicrobiomeProfile`
- `DailyScoreBreakdown`
- `PreparationModifier`
- `WeeklyDiversitySnapshot`
- et tout type/interface consommé à la fois par core et web

`@thirty/shared` ajoute `zod` comme dépendance pour la validation des schemas partagés.

#### 2. Adapters Prisma migrent vers `@thirty/core/infrastructure/`

Les implémentations Prisma des repositories quittent `apps/api/` pour être co-localisées par domaine dans core :

```
packages/core/src/
├── domains/
│   ├── scoring/
│   │   ├── repositories/       # ports (interfaces)
│   │   └── infrastructure/     # adapters Prisma
│   ├── journal/
│   │   ├── repositories/
│   │   └── infrastructure/
│   ├── diversity/
│   │   ├── repositories/
│   │   └── infrastructure/
│   └── suggestion/
│       ├── repositories/
│       └── infrastructure/
└── infrastructure/
    └── repositories/           # repos in-memory pour tests
```

#### 3. Nouveau package `@thirty/db` centralise Prisma

```
packages/db/
├── prisma/
│   ├── schema.prisma
│   └── migrations/
├── src/
│   ├── client.ts               # export du PrismaClient configuré
│   └── seed.ts                 # seed unifié (absorbe packages/food-db)
└── package.json
```

- Absorbe `apps/api/prisma/` (schema, migrations)
- Absorbe `packages/food-db/` (données de seed)
- Le client est généré dans `packages/db/src/generated/prisma`
- `packages/food-db` est supprimé

### Conséquences

- **Nouveau graphe de dépendances** : `shared` (leaf) <- `db` <- `core` <- `api`, `web` <- `shared` uniquement
- `@thirty/core` acquiert une dépendance sur `@thirty/db` (pour les adapters Prisma)
- `@thirty/shared` acquiert `zod` comme dépendance
- `apps/api` n'a plus de dossier `prisma/` ni de client généré localement
- `packages/food-db` est supprimé, son contenu absorbé par `@thirty/db`
- Les tests unitaires de core continuent d'utiliser les repos in-memory - les adapters Prisma sont testés en intégration
- L'ADR-008 reste valide pour l'organisation DDD par domaine dans core - seuls les emplacements des types, adapters et schema Prisma changent
