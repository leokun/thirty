## ADR-006 : Better Auth vs Passport.js

**Date** : 2026-04-02
**Statut** : Accepté

### Contexte

Thirty a besoin d'une couche d'authentification : email/password au minimum, OAuth Google souhaitable, sessions persistantes. NestJS propose officiellement l'intégration Passport.js via `@nestjs/passport`. Better Auth est une alternative plus récente, framework complet d'auth, avec une intégration NestJS communautaire.

### Options évaluées

**Option A — Passport.js (`@nestjs/passport`)**
- (+) Intégration officielle NestJS, très bien documentée
- (+) Standard de facto depuis 12+ ans dans l'écosystème Node.js
- (+) Contrôle total : chaque strategy, guard, et token est écrit à la main
- (+) Montre une compréhension profonde de l'auth (signal recruteur)
- (-) Verbeux : il faut implémenter LocalStrategy, JwtStrategy, GoogleStrategy, RefreshTokenGuard, le service de hashage, le stockage des refresh tokens, le reset password, la vérification email…
- (-) Passport est un middleware, pas une solution : il gère la vérification d'identité, pas le cycle de vie complet (signup, email verification, password reset, session management)
- (-) Estimation : 2-4 jours de travail pour un auth complet et robuste
- (-) Risque de bugs de sécurité dans le code custom (refresh token rotation, timing attacks sur la comparaison de hash…)

**Option B — Better Auth (`@thallesp/nestjs-better-auth`)**
- (+) Solution complète : signup, login, OAuth, sessions, refresh, email verification, password reset — tout est inclus
- (+) Self-hosted : les données utilisateur restent dans notre base PostgreSQL (cohérent avec le domaine santé)
- (+) Intégration Prisma native (adapter officiel)
- (+) AuthGuard global par défaut avec décorateurs `@AllowAnonymous()`, `@OptionalAuth()`, `@Session()` — API clean
- (+) Estimation : ~1 jour pour un auth complet
- (+) Sécurité auditée par la communauté plutôt que roulée à la main
- (-) Intégration NestJS communautaire (pas officielle NestJS)
- (-) Projet plus jeune (~2 ans vs 12 ans pour Passport)
- (-) Nécessite de désactiver le body parser NestJS natif (`bodyParser: false` dans `NestFactory.create`)
- (-) Moins de contenu éducatif / Stack Overflow disponible en cas de problème

### Décision

**Better Auth.**

L'auth n'est pas la feature différenciante de Thirty — c'est le scoring microbiote. Chaque jour passé à câbler du refresh token avec Passport est un jour de moins sur le produit. Better Auth est un choix pragmatique de lead : identifier ce qui est commodity et le déléguer à un outil fiable.

Le fait que Better Auth soit self-hosted est un avantage dans le contexte d'une app santé/nutrition : les données utilisateur (email, préférences alimentaires) restent dans notre PostgreSQL, pas chez un tiers. C'est un argument produit autant que technique.

La contrainte du body parser désactivé est un compromis acceptable — c'est documenté et isolé au bootstrap de l'app.

### Conséquences

- Le `NestFactory.create` utilise `{ bodyParser: false }` — Better Auth gère le parsing pour ses routes
- Le modèle User est géré par Better Auth via l'adapter Prisma — on étend avec nos champs custom (préférences, timezone)
- L'AuthGuard est global par défaut — les routes publiques utilisent `@AllowAnonymous()`
- OAuth Google est configurable day one via la config Better Auth (pas de GoogleStrategy à écrire)
- Si Better Auth pose des problèmes bloquants en production, la migration vers Passport est faisable (le modèle User reste dans Prisma, seule la couche auth change)
